## 2. Critical defect found during the audit (not mock-related)

Four backend files in the working tree were **silently truncated** (byte-for-byte prefixes
of their committed versions — a local file-sync fault, not a git operation):

- `Backend/app/core/config.py` (ended mid-`raise`, syntax error)
- `Backend/app/evidence_verification/__init__.py`
- `Backend/app/evidence_verification/providers.py` (lost ~700 lines)
- `Backend/app/evidence_verification/service.py`
- `Backend/tests/test_evidence_verification.py` (lost its last test)

All five were restored from `HEAD` and verified to compile. **Recommendation:** commit
immediately and investigate the sync tool (files were also truncated mid-session while
this audit ran). The remaining working-tree diff vs HEAD is CRLF/LF line-ending churn only.

## 3. Test fix (root cause, no test deleted)

`Backend/tests/test_evidence_verification.py` was env-sensitive: with a real `.env`
present (Duffel/IndianAPI/Aviationstack keys), provider routing selected native adapters
instead of the generic HTTP provider the tests monkeypatch → 4 failures. Fixed by adding
an autouse fixture that neutralizes all provider URLs/keys per test, making routing
deterministic regardless of local `.env`. No assertions were weakened.

## 5. Live-data verification evidence (measured, not estimated)

- `GET /health` → `{"status":"healthy","app":"GL Guardian Backend","version":"0.1.0"}`
- `GET /health/detailed` → `database: connected`, `active_investigations: 0`
- `GET /api/v1/investigations` without JWT → **401** (auth enforced in production ✓)
- `GET /metrics` → live Prometheus exposition, **31 metric families**; `http_requests_total`
  counters show the deployed UI performing `POST /api/v1/auth/token` and reading
  `/investigations/stats/summary`, `/investigations`, `/agents/health`, `/analytics/trend`
  — i.e. **the production UI provably consumes live backend data**.
- Latency (80 sequential requests from an external client, health endpoints):
  median **220 ms**, avg **334 ms**, p95 **1.25 s**, p99 **1.26 s**, errors **0%**.
- UI TTFB (10 requests to `/dashboard`): avg **188 ms**, p95 **247 ms**, errors **0%**.
- These are smoke measurements from one client region — **not** a load test. The k6
  load-test results in `load-tests/reports/` (p95 14.3 ms, 25 VUs) are from a local run
  with stub agents and exclude LLM latency.

**Not measured** (do not quote numbers for these): cloud k6 stress/soak runs, CPU/RAM

## 7. Required environment variables (live data flow)

**UI (Vercel):** `NEXT_PUBLIC_API_BASE_URL` (points at the Railway API; without it the UI
throws `ApiUnavailableError` and shows friendly errors — it never falls back to fake data),
plus optional `NEXT_PUBLIC_API_TOKEN` or `NEXT_PUBLIC_API_USERNAME`/`NEXT_PUBLIC_API_PASSWORD`.

**Backend (Railway):** `DATABASE_URL` (Neon Postgres), `SECRET_KEY`, `ENV=production`
(forces `AUTH_REQUIRED=true`), `DEFAULT_LLM_PROVIDER` + that provider's API key,
`USE_REAL_AGENTS=true` for real LLM reasoning (otherwise deterministic stubs),
optional: `USE_CELERY`, `USE_REDIS_EVENTS`, `USE_EVENTSTORE`, provider benchmark keys
(Duffel, IndianAPI fuel, GSTINCheck, Aviationstack). Full list: `ENVIRONMENT_VARIABLES.md`.
